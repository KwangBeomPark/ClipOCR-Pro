# myAGENT.md

## 1. 필수 워크플로우 (Mandatory Workflow)

AI 에이전트는 코드 작성 또는 수정 전에 반드시 아래 순서를 따른다.

1. **규칙 숙지**

   * `myAGENT.md` 및 `AI_CODE_MAP.md`를 먼저 읽고 작업 맥락을 파악한다.
   * 전역 규칙 보존을 위해 사용자의 명시적 요청이 없는 한 `myAGENT.md`는 임의로 수정하지 않는다.

2. **프로젝트 분석**

   * 현재 프로젝트의 폴더 구조, 주요 파일, 실행 흐름, 의존성을 파악한다.

3. **코드 지도 검토**

   * `AI_CODE_MAP.md`를 확인한다.
   * 수정 방지와 토큰 절약을 위해 정확한 라인 번호보다 파일명, 클래스명, 함수명, 주요 검색 키워드 중심으로 기록한다.

4. **코드 수정**

   * 기존 기능 보존 및 리팩토링 최소화 원칙을 준수하여 코드를 변경한다.
   * 사용자가 요청하지 않은 기능 추가, 단축키 변경, 변수명 변경, 구조 변경은 하지 않는다.

5. **코드 지도 갱신**

   * 모든 코드 수정과 자체 검증이 완료된 최종 단계에서만 `AI_CODE_MAP.md`를 갱신한다.
   * 파일 구조, 함수명, 핵심 변수, 설정 경로, 의존성, 단축키, 주요 기능 위치가 바뀐 경우에만 갱신한다.

6. **GitHub 공개 전 점검**

   * 공개 저장소 업로드 전에는 전체 파일 목록, `.gitignore`, README 링크, 이미지/GIF, 배포 바이너리 포함 여부를 확인한다.
   * 회사 PC에서 GitHub 웹 업로드를 직접 수행하는 경우 `.gitignore`가 자동 적용되지 않을 수 있으므로 업로드 파일 목록을 수동으로 대조한다.
   * `*.exe`, `*.zip`, 작업용 `tools/`, 개인 절대 경로가 포함된 파일은 저장소 추적 대상에서 제외하고 필요 시 GitHub Releases 또는 별도 로컬 보관 대상으로 분리한다.

## 2. 자체 검증 루프 (Verification Loop)

코드를 수정한 뒤 즉시 코드만 출력하는 것을 금지한다.
반드시 아래 4단계 요약을 제공한다.

1. **Plan**

   * 요구사항 요약
   * 수정 대상
   * 예상 리스크 및 사이드 이펙트

2. **Draft**

   * 초안에서 반영한 핵심 변경 사항 요약

3. **Review**

   * 문법 오류 가능성
   * 실행 오류 가능성
   * 엣지 케이스
   * 경로, 권한, 인코딩 문제
   * 기존 기능 영향 여부

4. **Final Code**

   * 검토 결과를 반영한 최종 코드 또는 패치
   * 변경 파일 목록
   * `AI_CODE_MAP.md` 갱신 여부
   * 남은 수동 테스트 항목

내부 사고 과정 전체를 노출하지 말고, 업무용 요약 리포트 형태로 간결하게 작성한다.

## 3. 전역 코딩 규칙 (Global Coding Rules)

* **한글 소통**

  * 사용자와의 대화, 설명, 작업 계획, 검증 요약, 변경 내역은 한국어를 기본으로 한다.
  * 코드, 함수명, 변수명, 파일명, 명령어, 경로, 에러 메시지 원문은 필요한 경우 영어 또는 원문을 유지한다.
  * 공개용 소스 코드의 주석은 영어를 기본으로 작성한다.
  * 변수명, 함수명, 클래스명, 설정 키, Registry value name은 영어 식별자를 유지한다.
  * 외부 라이브러리 파일은 원본성과 호환성을 우선하며, 주석이나 식별자를 임의로 번역하지 않는다.

* **안전한 수정**

  * 기존 코드를 먼저 읽고, 소규모 단위로 보수적으로 수정한다.
  * 기존 기능을 깨지 않는 변경을 우선한다.
  * 오류를 조용히 무시하지 않고, 원인 파악이 가능한 메시지나 로그를 남긴다.

* **리팩토링 제한**

  * 사용자가 명시적으로 요청하지 않은 대규모 구조 변경은 절대 하지 않는다.
  * 기존 단축키, 함수명, 변수명, 파일명, 설정 키, 영문 주석, 기존 UI 문구는 안정성을 위해 불필요하게 번역하거나 변경하지 않는다.
  * 구조 변경이나 네이밍 수정이 불가피하게 필요한 경우, 기존 기능을 깨지 않는 선에서 보수적으로 접근하며 변경 사유와 영향 범위를 먼저 요약하고 관련 문서와 제외 규칙을 함께 수정한다.

* **하드코딩 지양**

  * 설정값, 경로, 매직 넘버는 가능한 상수 또는 설정 영역으로 분리한다.
  * 사용자별 절대 경로를 코드에 직접 고정하지 않는다.
  * 임시 파일 경로는 앱 전용 폴더를 사용한다.

* **공개 저장소 보안**

  * 업로드 전 `password`, `token`, `secret`, `api key`, `authorization`, `bearer`, 사용자 프로필 절대 경로, 로컬 file URL, 네트워크 공유 경로, 회사 내부 시스템명, 개인정보 키워드를 검색한다.
  * 공개 유지가 필요한 작성자 GitHub 링크, 후원 링크, 라이선스 저작권 표기는 민감정보와 구분하여 문서에 명시한다.
  * 화면 캡처 이미지와 GIF는 텍스트 검색으로 확인되지 않으므로 대표 프레임을 직접 확인한다.
  * 실행 파일과 압축 파일은 소스 저장소가 아니라 GitHub Releases에 올리는 방식을 우선한다.

* **UI 기본 방침**

  * 창 위치가 명시되지 않으면 커서 근처 또는 화면 중앙에 열리도록 계산한다.
  * 플로팅 창이나 AlwaysOnTop 창이 있는 경우 신규 대화상자, InputBox, 메시지 박스가 가려지지 않도록 Z-Order를 검토한다.
  * 멀티 모니터, DPI scaling, 화면 경계 밖 표시 가능성을 고려한다.

## 4. 기업용 Windows 환경 대응 (Corporate Windows Rules)

* 관리자 권한 없이 동작하는 Portable 방식을 우선한다.
* 무거운 외부 라이브러리나 DLL 의존성을 무분별하게 추가하지 않는다.
* 회사 보안 정책상 차단될 수 있는 방식은 피한다.
* 파일 경로에 공백, 한글, 영어, 폴란드어 문자가 포함되어도 정상 동작해야 한다.
* 파일 잠김, 네트워크 드라이브 연결 실패, OneDrive/SharePoint 동기화 지연을 고려한다.
* 파일, 레지스트리, 클립보드, 외부 프로세스 접근에는 예외 처리를 적용한다.
* 삭제 작업은 반드시 앱이 소유한 임시 폴더나 명확히 지정된 대상에만 수행한다.

## 5. 프로그래밍 언어별 특화 규칙 (Language-Specific Rules)

아래 규칙은 해당 언어가 현재 프로젝트에서 사용될 때 적용한다.

### AutoHotkey v2 규칙

* AutoHotkey v2 문법만 사용하고 v1 문법을 혼용하지 않는다.
* 함수 내부에서 외부 전역 변수를 수정하거나 접근할 때는 v2 규칙에 따라 `global` 선언을 명확히 한다.
* 단축키 정의, 메뉴 생성, 설정 로드, 액션 실행 로직을 가능한 분리한다.
* 신규 GUI 인스턴스 생성 전, 기존 동일 목적의 GUI 객체가 있으면 명시적으로 해제한다.
* GUI 작업 시 DPI scaling, 멀티 모니터, 화면 밖 표시 가능성을 검토한다.
* `AlwaysOnTop` 또는 플로팅 창 하위에서 대화상자가 가려질 수 있으면 `+OwnDialogs` 등 적절한 창 소유 관계를 검토한다.
* GDI+를 사용하는 경우 작업 완료 후 이미지, 그래픽, 핸들 등 관련 자원을 명시적으로 해제한다.
* 클립보드, 파일, 레지스트리 읽기/쓰기에는 `try-catch`와 안전한 fallback을 적용한다.
* 파괴적 변경이나 덮어쓰기 작업에는 백업 또는 복구 가능성을 우선 고려한다.

## 6. 산출물 및 빌드 관리 원칙 (Artifacts & Build Management)

* **소스 루트 폴더 구성**
  * 프로젝트 소스 루트에는 사람이 직접 관리하는 원본 소스 코드, 문서(README 등), 공개용 이미지 자산만 위치해야 한다.
  * 실행 파일(`.exe`), 압축 파일(`.zip`), 설치 파일, 패키징 결과물 등은 루트에 두지 않고 `dist/`, `release/`, `build/`, `out/` 같은 별도 산출물 폴더에 모아 관리한다.
  * 위 산출물 폴더들은 기본적으로 Git 추적 대상이 되지 않도록 `.gitignore`에 명시적으로 등록한다.

* **빌드 스크립트 분리**
  * 빌드/패키징을 위한 스크립트가 필요한 경우, 루트가 아닌 `scripts/` 폴더에 분리하여 보관하며 스크립트 내 산출물 출력 경로는 명확히 지정한다.

* **릴리즈 배포**
  * 공개 배포용 바이너리(`.exe`, `.zip`, installer 등)는 저장소(repository) 파일로 커밋하지 않으며, GitHub Releases 같은 릴리즈 자산으로 업로드하는 것을 원칙으로 한다.
  * `README.md` 문서에는 "소스 코드로 실행하는 방법"과 "릴리즈에서 다운로드하여 사용하는 방법"을 명확히 구분하여 안내한다.

* **배포 전 민감 정보 검사**
  * 공개 업로드 전, 산출물 폴더 내 패키징된 결과물에 개인 설정, 로그 파일, 회사 내부 경로, 사용자 데이터, 비밀번호, API Key, Token 등의 민감 정보가 섞여 들어가지 않았는지 반드시 점검한다.
